# Restaurant hero-image generation pipeline

Date: 2026-06-15
Status: Approved design, pending spec review
Branch: feat/functional-mobile-mvp

## Problem

Restaurant cover art is currently a deterministic cuisine gradient plus a monogram
(for example "CF", "MC"). We want each restaurant we onboard to get a real,
AI-generated, food-forward cover image themed to its cuisine, stored durably and
shown automatically wherever the restaurant appears.

## Decisions (from brainstorming)

- **Storage:** Supabase Storage (public bucket). The public URL goes in the
  existing `restaurants.hero_image` column. No repo bloat, scales to dynamic
  onboarding, one source for every surface that reads the DB.
- **Scope:** Pipeline only. We do NOT batch-generate the 200 seed restaurants
  (they are fictional demo data generated by `frontend/scripts/genSeed.mjs`).
  We build a reusable generator and run it per restaurant as real ones are onboarded.
- **Style:** Food-forward photography themed per cuisine, in a consistent warm,
  appetizing style that matches the existing `website/img/hero-*.jpg` images.
- **Uniqueness:** One unique image per restaurant (not one shared image per cuisine).

## Why a Node script (not an Edge Function or the Express backend)

Findings from the codebase:

- There is no automated "onboard a restaurant" event today. `restaurant_claims`
  is write-only intent capture; nothing converts a claim into a `restaurants` row
  or flips `is_partner`. The Admin page (`frontend/src/pages/Admin.tsx`) is
  read-only analytics.
- The only code that writes the `restaurants` table is the Node seed-migration
  script `frontend/scripts/migrateToSupabase.mjs`, which reads
  `SUPABASE_SERVICE_ROLE_KEY` from `frontend/.env`.
- The Express `backend/` exists but the live app talks to Supabase directly, so it
  is effectively dormant. No Supabase Edge Functions, no serverless routes.

Generation needs secret keys (kie.ai + Supabase service-role), so it must run
server-side. The lowest-friction, pattern-consistent home is a Node script that
mirrors `migrateToSupabase.mjs`. Its core is written as a pure, injectable function
so it can later be lifted into a Supabase Edge Function (the documented upgrade
path) once a real onboard trigger exists.

## Architecture

A reusable generator with a thin CLI, under `frontend/scripts/`:

```
frontend/scripts/
  generateHeroImage.mjs            # CLI wrapper: env + arg parsing, calls the pipeline
  lib/
    heroPrompt.mjs                 # pure: (cuisine, {vibe, name}) -> prompt string
    kieImage.mjs                   # createTask + poll + extract result URL
    heroImagePipeline.mjs          # orchestration core (injectable deps), no process exit
    heroPrompt.test.mjs            # unit tests
    kieImage.test.mjs              # unit tests (URL extraction from malformed JSON)
```

Separation of concerns so each unit is understandable and testable on its own, and
so `heroImagePipeline.mjs` can be imported by a future Edge Function unchanged.

### Components

- **`heroPrompt(cuisine, { vibe, name })` (pure).** Returns a food-forward prompt
  built from a per-cuisine subject map plus a shared style suffix. Unknown cuisines
  fall back to a generic appetizing-dish subject. Optional `vibe` (from the
  restaurant's `vibe` array) is appended lightly as ambiance flavour. Deterministic
  and side-effect free.

- **`kieImage({ prompt, aspectRatio, apiKey, fetch })`.** Calls kie.ai:
  - `POST https://api.kie.ai/api/v1/jobs/createTask` with
    `{ model, input: { prompt, aspect_ratio } }` -> returns a `taskId`.
  - Polls `GET https://api.kie.ai/api/v1/jobs/recordInfo?taskId=...` until
    `data.state` is `success` or `fail` (or timeout).
  - Extracts the result image URL from `data.resultJson`. The kie response JSON is
    malformed (double-encoded `param`), so the URL is extracted by regex
    (`https://[^"]*\.(png|jpe?g)`) rather than `JSON.parse`.
  - Model: reuse the exact kie.ai GPT-image model/flow that produced the current
    `hero-*.jpg` images (recorded as `gpt-image-2-text-to-image`; confirm against
    kie.ai at implementation time). `aspect_ratio`: `3:2` (landscape, matches the
    card and detail-hero crops).

- **`download(url, { fetch })`.** Pulls the image through the `images.weserv.nl`
  proxy, which both bypasses the direct kie CDN connection resets seen in this
  environment and resizes/optimizes. Request `?url=<host/path>&w=1200&output=jpg&q=82`.
  Returns a JPEG buffer (~1200px wide). Retry once, then fall back to a direct fetch.

- **`storage` (Supabase).** Ensure a public bucket `restaurant-images` exists
  (`createBucket('restaurant-images', { public: true })`, ignore "already exists").
  Upload `<slug>.jpg` with `{ contentType: 'image/jpeg', upsert: true }`. Return
  `getPublicUrl('<slug>.jpg')`.

- **`db` (Supabase).** `update({ hero_image: url }).eq('slug', slug)` on
  `restaurants`, using the service-role client.

- **`heroImagePipeline(restaurant, deps)` (core).** Orchestrates:
  skip-if-already-set (unless force) -> `heroPrompt` -> `kieImage` -> `download`
  -> `storage.upload` -> `db.update`. Takes injected `{ supabase, fetch, kieApiKey,
  force }` so it is unit-testable and reusable by a future trigger. Returns
  `{ slug, url, skipped }`.

- **`generateHeroImage.mjs` (CLI).** Parses `frontend/.env` (reusing the small
  parser already in `migrateToSupabase.mjs`), builds the service-role Supabase
  client, parses args, and calls the pipeline. Modes:
  - `node scripts/generateHeroImage.mjs <slug>` - one restaurant (reads its
    cuisine/vibe/name from the DB).
  - `--missing` - every restaurant where `hero_image` is null (backfill; available
    but not run as part of this work).
  - `--force` - regenerate even if `hero_image` is already set.
  - `--dry-run` - print the prompt only; no API calls, no spend.

## Data flow

```
onboard a restaurant (manual today; programmatic later)
  -> generateHeroImage(slug)
     -> heroPrompt(cuisine, {vibe,name})
     -> kieImage(prompt)               (kie.ai create + poll)
     -> download(resultUrl)            (weserv proxy -> JPEG buffer)
     -> storage.upload(<slug>.jpg)     (Supabase public bucket)
     -> db.update(hero_image = url)     (restaurants table)
  -> app RestaurantCover renders <img src={heroImage}>   (already implemented)
```

## Rendering

- **App:** `frontend/src/components/RestaurantCover.tsx` already renders
  `restaurant.heroImage` when present and falls back to the gradient otherwise.
  Required check: confirm `rowToRestaurant` in `frontend/src/lib/api.ts` maps
  `hero_image` -> `heroImage`. If it does not, add that one mapping (without it the
  column is set but the cover will not show).
- **Website:** Out of scope here. The website builds from the fictional seed and
  does not read the DB, so real onboarded restaurants do not appear on it at all
  yet. Showing DB-backed restaurants and their images on the website is a separate
  effort (see Follow-ups).

## Secrets and configuration

- Add `KIE_API_KEY` to `frontend/.env` (gitignored) and a name-only entry in
  `frontend/.env.example`.
- Reuse the existing `SUPABASE_SERVICE_ROLE_KEY` and `VITE_SUPABASE_URL` from
  `frontend/.env`.
- The kie.ai key pasted in chat earlier should be rotated; the new key goes only in
  `.env`, never in the repo or chat. The key is not currently on disk and must be
  re-supplied before the live proof.

## Idempotency and error handling

- Skip when `hero_image` is already set unless `--force`. Storage upload uses
  `upsert: true` so a forced regenerate overwrites cleanly.
- `kieImage`: throw on create failure; poll timeout after ~3 minutes
  (60 attempts x 3s); throw on `state: fail` with the reason; throw if no URL can
  be extracted.
- `download`: retry once via weserv, then one direct-fetch fallback.
- `--missing` batch: catch per restaurant, log, continue, and print a
  success/failure summary at the end. A single failure never aborts the batch.
- `migrateToSupabase.mjs` does NOT send `hero_image` in its upsert payload, so
  re-seeding never clobbers generated images (PostgREST merge-duplicates updates
  only the columns provided). Add a code comment making this explicit.

## Prompt design

Shared style suffix (applied to every prompt for cohesion):

> Professional food photography, 45-degree angle, warm natural window light,
> shallow depth of field, fresh and appetizing, rustic table setting, vibrant but
> natural colours, editorial restaurant-magazine style. No text, no logos, no
> watermarks, no people, no hands.

Per-cuisine subject map (covers every cuisine in the seed; `default` for any new
value):

| cuisine | subject |
| --- | --- |
| Spanish tapas | a colourful spread of Spanish tapas: patatas bravas, jamon iberico, garlic prawns, croquetas and olives on small ceramic plates |
| paella | an authentic Valencian seafood paella in a traditional pan with mussels, prawns, lemon wedges and saffron rice |
| menu del dia | a hearty Spanish menu-del-dia plate: grilled meat, rustic vegetables and bread on a simple ceramic plate |
| sushi | an elegant assortment of fresh sushi nigiri and sashimi on a dark slate board with soy sauce and wasabi |
| burgers | a gourmet cheeseburger with melted cheese, fresh lettuce and tomato, with crispy fries on a wooden board |
| pizza | a wood-fired Neapolitan margherita pizza with bubbling mozzarella, fresh basil and tomato on a rustic surface |
| pasta | a bowl of fresh handmade pasta with rich tomato sauce, parmesan and basil |
| healthy bowls | a vibrant healthy grain bowl with avocado, roasted vegetables, chickpeas and greens |
| vegan | a colourful vegan buddha bowl with roasted vegetables, tofu, quinoa and tahini dressing |
| vegetarian | a fresh vegetarian dish with grilled seasonal vegetables, herbs and burrata |
| brunch | a brunch spread with avocado toast, poached eggs, pancakes and fresh orange juice |
| coffee | a beautifully crafted flat white with latte art beside a slice of cake on a cafe table |
| bar | craft cocktails and a sharing board of snacks on a moody bar counter |
| Mexican | authentic Mexican tacos with fresh cilantro, onion, lime and salsa on a colourful plate |
| Indian | a rich Indian curry with naan bread, basmati rice and fresh herbs in copper bowls |
| Asian fusion | a modern Asian-fusion plate of bao buns and dumplings with dipping sauces on slate |
| Mediterranean | a Mediterranean mezze platter with hummus, grilled vegetables, olives and pita |
| seafood | a fresh seafood platter with grilled fish, prawns, mussels and lemon on ice |
| steak | a grilled ribeye steak with rosemary, sliced to show a juicy medium-rare interior, on a cast-iron skillet |
| default | an appetising signature dish, beautifully plated |

Map keys must match the exact seed `cuisine` strings, including accents (for
example `menú del día`). The readable table above drops accents for legibility;
the implementation keys on the literal values.

## Testing

- **Unit (vitest):**
  - `heroPrompt`: returns the right subject per cuisine, includes the style suffix,
    falls back to `default` for unknown cuisines, and incorporates `vibe` when given.
  - `kieImage` URL extraction: given a sample malformed kie `resultJson`, returns
    the correct image URL; throws when none is present.
  - Ensure vitest picks up `frontend/scripts/lib/*.test.mjs` (extend the test
    include glob if needed).
- **Live proof (one image, gated on spend approval and the kie key):** run the
  generator for a single existing DB restaurant, confirm `hero_image` is set, the
  public URL returns 200 image bytes, and the app's `RestaurantCover` renders it.

## Out of scope / follow-ups

- Batch-generating the 200 fictional seed restaurants.
- Website reading restaurants from Supabase and rendering `heroImage` (prerequisite
  for any generated image to appear on foodmatch.es).
- Supabase Edge Function auto-trigger reusing `heroImagePipeline.mjs`, once a real
  onboard event exists (claim approval, admin "add restaurant", or a verified flag).
- The onboarding flow itself (claim approval -> create/verify restaurant row),
  which is a separate feature.

## Risks and open questions

- Exact kie.ai model id to reuse: confirm it matches the one that produced the
  current hero images before spending.
- kie.ai cost per image: confirm before any batch (not batching now).
- Public bucket read is acceptable (cover images are public content by nature).
